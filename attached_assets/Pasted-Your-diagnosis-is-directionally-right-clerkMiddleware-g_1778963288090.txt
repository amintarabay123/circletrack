Why hardcoding publishableKey: LIVE_PK may still have failed
1. Deploy / code path didn’t actually change (very common)
Replit didn’t redeploy, wrong branch, or old process still running.
Change was in a file that isn’t what production runs.
Check: Log at startup: which publishableKey prefix the server uses (pk_live vs pk_test).

2. Only publishableKey was fixed; secretKey pairing still wrong
@clerk/express uses CLERK_SECRET_KEY for verification. You need both from the same Clerk app:

publishableKey = pk_live_…
secretKey = sk_live_… (matching that app)
Passing live publishable + sk_live is correct if they’re the same Clerk application. Passing live publishable + env still exposing pk_test elsewhere can still break things.

Current repo: app.ts still has plain clerkMiddleware() with no explicit keys — it uses env only.

3. Second mismatch: /api/config overwrites the app’s Clerk key (high probability)
On launch, _layout.tsx does:

GET https://<EXPO_PUBLIC_DOMAIN>/api/config
→ uses cfg.publishableKey (server’s CLERK_PUBLISHABLE_KEY = pk_test)
→ ClerkProvider(publishableKey: pk_test, proxyUrl: …)
So even a TestFlight build baked with pk_live can switch to pk_test at runtime.

routes/config.ts returns:

publishableKey from CLERK_PUBLISHABLE_KEY (Replit = pk_test)
proxyUrl → https://…/api/__clerk when CLERK_PROXY_URL is set
That’s a client-side reconfiguration, independent of clerkMiddleware({ publishableKey: LIVE_PK }).

4. Clerk proxy: live secret + test publishable key (high probability on Replit prod)
In production, clerkProxyMiddleware sends Clerk-Secret-Key: CLERK_SECRET_KEY (sk_live_…) to Clerk’s FAPI, while the app may use pk_test_… from /api/config.

So sign-in can “work” in the UI but tokens may not match what API middleware expects.

Symptom: SSO OK, API 401 — matches your report.

5. getToken() is probably fine
getToken() from @clerk/clerk-expo normally returns a session JWT suitable for Authorization: Bearer. Unlikely the root cause unless:

token is null (you’d often see intermittent 401, not always)
wrong template / org claim (less common here)
Bearer is attached per your note — so the issue is verification, not missing header.

6. eas.json in repo may not match what you think
In the repo on disk, both preview and production use:

EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY = pk_test_…

not pk_live_…. If TestFlight was built from this repo without changing eas.json, the client is also on test, and the story is “everything test except sk_live on server” — still inconsistent.

Worth confirming which key the actual TestFlight build was made with (EAS build logs).
